TraceGuard has a clean adapter architecture. Each integration lives in its own
folder under `integrations/` and connects to the **local daemon** — none of them
require the cloud.

| Integration | Path | Primary mechanism |
| --- | --- | --- |
| [GitHub](/integrations/github) | `integrations/github` | GitHub App + Actions, sanitized check summaries |
| [Claude Code](/integrations/claude) | `integrations/claude` | Hooks adapter, wrapper fallback |
| [Codex](/integrations/codex) | `integrations/codex` | CLI wrapper adapter |
| [VS Code](/integrations/vscode) | `integrations/vscode` | Extension bridging to the daemon |
| [Cursor](/integrations/cursor) | `integrations/cursor` | MCP server exposing TraceGuard tools |

## Honest boundaries

- Terminal agents launched through `trg run` are wrapped directly and fully
  attributed.
- GUI tools are observed via file changes and Git diffs. **Full command
  guarding requires supported hooks/integration points or running through the
  wrapper.**
- Integrations send only sanitized summaries off-machine (GitHub). They never
  upload raw project files, raw secrets, or the local SQLite database.

## "Enterprise-level" in the MVP

For now this means a clean integration architecture, documented adapters, secure
local daemon, and sanitized summaries — **not** cloud-first. Org policy files,
central policy management, team dashboards, SSO, and audit exports are explicitly
out of scope until requested.
